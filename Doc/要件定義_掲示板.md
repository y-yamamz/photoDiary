# PhotoDiary 掲示板機能 要件定義

---

# 1. 機能概要

ユーザーと管理者間のコミュニケーション機能。

| 方向 | 内容 |
|------|------|
| 管理者 → 全ユーザー | お知らせを投稿し、全ユーザーの掲示板に表示する |
| ユーザー → 管理者 | 問い合わせを送信する（1対1） |
| 管理者 → 返信 | 問い合わせに対し「個別返信」または「全体お知らせとして返信」を選択して返信し、対応済みにする |

---

# 2. 確定要件

## 2.1 認証方式

- 管理者操作はすべて既存の「シークレットキー」入力方式を使用する

## 2.2 データ取得タイミング

- リアルタイム通知（WebSocket）は実装しない
- お知らせ・問い合わせはページを開いたタイミングでAPIを呼び出し取得する

## 2.3 返信機能

- 管理者は問い合わせに対して返信できる
- 返信時に配信先を選択する：
  - **個別返信**：問い合わせユーザーのみの問い合わせスレッドに表示される
  - **全体お知らせとして返信**：notices テーブルに新規レコードを追加し、全ユーザーのお知らせ一覧に表示される
- 返信送信後、対象の問い合わせのステータスを「対応済み」に更新する

## 2.4 状態管理

- 問い合わせのステータス：`0 = 未対応` / `1 = 対応済み`
- お知らせの既読管理：notice_reads テーブルで管理（ユーザーがお知らせ一覧を開いた時に未読→既読に更新）

# 4. テーブル設計

## 4.1 notices（お知らせ）

管理者が全ユーザーへ向けて投稿するお知らせ。
問い合わせ返信で「全体お知らせとして投稿」を選択した場合も本テーブルに INSERT される。

| カラム名       | 型            | PK | NN | デフォルト                | コメント                        |
|--------------|-------------|----|----|------------------------|-------------------------------|
| notice_id    | BIGINT       | ○  | ○  | AUTO_INCREMENT         | お知らせID                      |
| title        | VARCHAR(255) |    | ○  |                        | タイトル                        |
| body         | TEXT         |    | ○  |                        | 本文                            |
| created_at   | DATETIME     |    | ○  | CURRENT_TIMESTAMP      | 作成日時                        |
| updated_at   | DATETIME     |    |    | ON UPDATE CURRENT_TIMESTAMP | 更新日時                   |

---

## 4.2 notice_reads（お知らせ既読管理）

ユーザーごとのお知らせ既読状態を管理する。
ユーザーがお知らせ一覧を開いた際に、未読レコードを INSERT（既読化）する。

| カラム名       | 型        | PK | NN | デフォルト           | コメント               |
|--------------|---------|----|----|-------------------|--------------------|
| notice_id    | BIGINT   | ○  | ○  |                   | お知らせID（FK: notices） |
| user_id      | BIGINT   | ○  | ○  |                   | ユーザーID（FK: users）   |
| read_at      | DATETIME |    | ○  | CURRENT_TIMESTAMP | 既読日時               |

**複合主キー**：(notice_id, user_id)

---

## 4.3 inquiries（問い合わせ）

ユーザーから管理者へ送信される問い合わせ。

| カラム名       | 型            | PK | NN | デフォルト                | コメント                       |
|--------------|-------------|----|----|------------------------|------------------------------|
| inquiry_id   | BIGINT       | ○  | ○  | AUTO_INCREMENT         | 問い合わせID                   |
| user_id      | BIGINT       |    | ○  |                        | 送信ユーザーID（FK: users）      |
| subject      | VARCHAR(255) |    | ○  |                        | 件名                           |
| body         | TEXT         |    | ○  |                        | 本文                           |
| status       | TINYINT      |    | ○  | 0                      | 状態（0=未対応, 1=対応済み）     |
| created_at   | DATETIME     |    | ○  | CURRENT_TIMESTAMP      | 送信日時                       |
| updated_at   | DATETIME     |    |    | ON UPDATE CURRENT_TIMESTAMP | 更新日時                  |

---

## 4.4 inquiry_replies（問い合わせ返信）

管理者が問い合わせに対して送信する返信。

| カラム名        | 型       | PK | NN | デフォルト           | コメント                                                |
|---------------|--------|----|----|-------------------|----------------------------------------------------|
| reply_id      | BIGINT  | ○  | ○  | AUTO_INCREMENT    | 返信ID                                              |
| inquiry_id    | BIGINT  |    | ○  |                   | 問い合わせID（FK: inquiries）                         |
| body          | TEXT    |    | ○  |                   | 返信本文                                            |
| target_type   | TINYINT |    | ○  | 0                 | 配信先（0=個別返信, 1=全体お知らせとして投稿）          |
| notice_id     | BIGINT  |    |    | NULL              | 全体投稿時に notices に INSERT した ID（FK: notices）  |
| created_at    | DATETIME|    | ○  | CURRENT_TIMESTAMP | 返信日時                                            |

---

# 5. テーブル関連図

```
users
  │
  ├── inquiries (user_id)
  │     └── inquiry_replies (inquiry_id)
  │               └── notices (notice_id) ← target_type=1 の場合のみ参照
  │
  └── notice_reads (user_id)
              └── notices (notice_id)

notices ←──── inquiry_replies (notice_id, NULL許容)
notices ←──── notice_reads (notice_id)
```

# 6. API設計

## 6.1 ユーザー向け API

| メソッド | URL                           | 内容                                               |
|--------|-------------------------------|--------------------------------------------------|
| GET    | /api/board/notices            | お知らせ一覧取得（未読フラグ付き）                     |
| POST   | /api/board/notices/read       | お知らせ既読登録（一覧表示時に一括既読化）             |
| GET    | /api/board/inquiries          | 自分の問い合わせ一覧取得（返信含む）                   |
| POST   | /api/board/inquiries          | 問い合わせ新規送信                                   |

## 6.2 管理者向け API

| メソッド | URL                                      | 内容                                          |
|--------|------------------------------------------|---------------------------------------------|
| GET    | /api/admin/board/notices                 | お知らせ一覧取得                               |
| POST   | /api/admin/board/notices                 | お知らせ新規投稿（全体）                        |
| PUT    | /api/admin/board/notices/{noticeId}      | お知らせ編集                                  |
| DELETE | /api/admin/board/notices/{noticeId}      | お知らせ削除                                  |
| GET    | /api/admin/board/inquiries               | 問い合わせ一覧取得（未対応件数含む）             |
| POST   | /api/admin/board/inquiries/{id}/reply    | 問い合わせへ返信（個別 or 全体選択）＋対応済み更新 |

> 管理者 API はすべてリクエストボディまたはヘッダーにシークレットキーを含める

## 06_create_notices.sql

```sql
-- ============================================================
-- NOTICES（お知らせ）テーブル作成
-- 依存 : なし
-- DB     : MySQL 8.0+
-- 文字コード : utf8mb4 / utf8mb4_unicode_ci
-- ============================================================

CREATE TABLE IF NOT EXISTS notices (
  notice_id  BIGINT        NOT NULL AUTO_INCREMENT  COMMENT 'お知らせID',
  title      VARCHAR(255)  NOT NULL                 COMMENT 'タイトル',
  body       TEXT          NOT NULL                 COMMENT '本文',
  created_at DATETIME      NOT NULL
             DEFAULT CURRENT_TIMESTAMP              COMMENT '作成日時',
  updated_at DATETIME               DEFAULT NULL
             ON UPDATE CURRENT_TIMESTAMP            COMMENT '更新日時',

  PRIMARY KEY (notice_id)

) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci
  COMMENT='お知らせ';
```

## 07_create_notice_reads.sql

```sql
-- ============================================================
-- NOTICE_READS（お知らせ既読管理）テーブル作成
-- 依存 : notices, users
-- DB     : MySQL 8.0+
-- 文字コード : utf8mb4 / utf8mb4_unicode_ci
-- ============================================================

CREATE TABLE IF NOT EXISTS notice_reads (
  notice_id BIGINT   NOT NULL  COMMENT 'お知らせID',
  user_id   BIGINT   NOT NULL  COMMENT 'ユーザーID',
  read_at   DATETIME NOT NULL
            DEFAULT CURRENT_TIMESTAMP  COMMENT '既読日時',

  PRIMARY KEY (notice_id, user_id),

  CONSTRAINT fk_notice_reads_notice_id
    FOREIGN KEY (notice_id) REFERENCES notices (notice_id)
    ON DELETE CASCADE
    ON UPDATE CASCADE,

  CONSTRAINT fk_notice_reads_user_id
    FOREIGN KEY (user_id) REFERENCES users (user_id)
    ON DELETE CASCADE
    ON UPDATE CASCADE

) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci
  COMMENT='お知らせ既読管理';
```

## 08_create_inquiries.sql

```sql
-- ============================================================
-- INQUIRIES（問い合わせ）テーブル作成
-- 依存 : users
-- DB     : MySQL 8.0+
-- 文字コード : utf8mb4 / utf8mb4_unicode_ci
-- ============================================================

CREATE TABLE IF NOT EXISTS inquiries (
  inquiry_id BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '問い合わせID',
  user_id    BIGINT        NOT NULL                 COMMENT '送信ユーザーID',
  subject    VARCHAR(255)  NOT NULL                 COMMENT '件名',
  body       TEXT          NOT NULL                 COMMENT '本文',
  status     TINYINT       NOT NULL DEFAULT 0       COMMENT '状態（0=未対応, 1=対応済み）',
  created_at DATETIME      NOT NULL
             DEFAULT CURRENT_TIMESTAMP              COMMENT '送信日時',
  updated_at DATETIME               DEFAULT NULL

  PRIMARY KEY (inquiry_id),

  INDEX idx_inquiries_user_id (user_id),
  INDEX idx_inquiries_status  (status),

  CONSTRAINT fk_inquiries_user_id
    FOREIGN KEY (user_id) REFERENCES users (user_id)
    ON DELETE CASCADE
    ON UPDATE CASCADE

) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci
  COMMENT='問い合わせ';
```

## 09_create_inquiry_replies.sql

```sql
-- ============================================================
-- INQUIRY_REPLIES（問い合わせ返信）テーブル作成
-- 依存 : inquiries, notices
-- DB     : MySQL 8.0+
-- 文字コード : utf8mb4 / utf8mb4_unicode_ci
-- ============================================================

CREATE TABLE IF NOT EXISTS inquiry_replies (
  reply_id    BIGINT   NOT NULL AUTO_INCREMENT  COMMENT '返信ID',
  inquiry_id  BIGINT   NOT NULL                 COMMENT '問い合わせID',
  body        TEXT     NOT NULL                 COMMENT '返信本文',
  target_type TINYINT  NOT NULL DEFAULT 0       COMMENT '配信先（0=個別返信, 1=全体お知らせ）',
  notice_id   BIGINT            DEFAULT NULL    COMMENT '全体投稿時のnotices ID（NULLは個別返信）',
  created_at  DATETIME NOT NULL
              DEFAULT CURRENT_TIMESTAMP         COMMENT '返信日時',

  PRIMARY KEY (reply_id),

  INDEX idx_inquiry_replies_inquiry_id (inquiry_id),

  CONSTRAINT fk_inquiry_replies_inquiry_id
    FOREIGN KEY (inquiry_id) REFERENCES inquiries (inquiry_id)
    ON DELETE CASCADE
    ON UPDATE CASCADE,

  CONSTRAINT fk_inquiry_replies_notice_id
    FOREIGN KEY (notice_id) REFERENCES notices (notice_id)
    ON DELETE SET NULL
    ON UPDATE CASCADE

) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci
  COMMENT='問い合わせ返信';
```

# 8. 処理フロー

## 8.1 管理者が「全体お知らせとして返信」する場合のトランザクション

```
① notices に INSERT（タイトル・本文）
   → notice_id を取得
② inquiry_replies に INSERT
   （inquiry_id, body, target_type=1, notice_id=①で取得したID）
③ inquiries の status を 1（対応済み）に UPDATE
```

上記 ①〜③ を1トランザクションで実行する。

## 8.2 管理者が「個別返信」する場合

```
① inquiry_replies に INSERT
   （inquiry_id, body, target_type=0, notice_id=NULL）
② inquiries の status を 1（対応済み）に UPDATE
```

## 8.3 ユーザーがお知らせ一覧を開く場合

```
① notices を全件取得
② notice_reads から自分の既読 notice_id を取得
③ 未読（notice_reads に存在しない）のものに NEW フラグを付与して返却
④ 未読 notice_id を notice_reads に一括 INSERT（既読化）
```

# 9. 補足・制約

- お知らせの削除時、紐づく notice_reads は CASCADE DELETE で自動削除
- 問い合わせユーザーが削除された場合、紐づく inquiries・inquiry_replies も CASCADE DELETE
- inquiry_replies.notice_id は notices 削除時に SET NULL（返信履歴は残す）
- 管理者シークレットキーは既存方式と同様にバックエンドで検証する
- 問い合わせ本文・返信本文の最大長は TEXT 型（65,535 bytes）に準拠
